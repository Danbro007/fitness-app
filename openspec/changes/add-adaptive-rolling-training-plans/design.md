## Context

应用当前使用 SQLite 9，已有场地、器械绑定、正式计划、计划动作、训练会话、组记录、用户档案、AI 草稿和训练调整表。现有 `generateWeeklyPlanDraft()` / `confirmFourWeekPlanDraft()` 以 AI 文本草稿为中心，无法表达计划周期、指定星期与场地、输入快照、逐动作解释和草稿过期，也会一次确认多周计划。

本变更跨越 persistence、domain、repository、Compose UI 和备份。必须保持五个主导航、本地优先、AI 可选、SQLite 为正式数据源、草稿经用户确认后落库，以及阻塞 I/O 离开主线程。

## Goals / Non-Goals

**Goals:**

- 建立可迁移、可备份的计划周期和周日程模型。
- 使用纯 Kotlin 规则完成趋势、重量档位、安全和可行性校验。
- 让 AI 只负责候选内容与解释，最终草稿必须经过本地规则校验。
- 支持输入快照、草稿过期、整周原子确认和逐周主动生成。
- 将训练反馈安全地反馈到下一周计划。

**Non-Goals:**

- 不自动调整目标组数或次数。
- 不精确预测训练分钟数。
- 不做医学诊断或保证伤病动作映射绝对正确。
- 不引入账号、云同步、服务端或后台自动 AI 调用。
- 不在一个 PR 中重写现有计划、训练或设置页面。

## Decisions

### 1. 复用正式计划表，新增周期与草稿表

确认后的训练继续写入 `planned_workout` 和 `planned_exercise`，避免复制训练执行依赖。新增：

- `plan_cycle`：周期范围、状态、当前周次和时间戳。
- `plan_schedule_day`：周期中的星期、场地和排序。
- `weekly_plan_draft`：目标周、结构化 payload、输入快照哈希、状态和解释。
- `action_preference`：动作长期包含/排除/替换偏好。
- `injury_filter_override`：伤病文本版本下经二次确认的动作例外。
- `venue_equipment_load`：场地器械的可用重量档位。

替代方案是继续把所有信息塞进 `ai_draft.content`。拒绝该方案，因为无法建立外键、一致查询、原子确认和可靠过期判断。

### 2. SQLite 9 升级到 10，备份版本 4 升级到 5

迁移只新增表和必要列，不删除或重写旧数据。版本 4 备份继续可导入；缺失新字段时使用空集合并保留原有数据。版本 5 导出新增计划周期相关数据。

### 3. 纯规则模块先于 repository 和 UI

新增无 Android 依赖的 `AdaptivePlanPolicy`、`PlanConstraintValidator` 和结构化输入/输出模型。JVM 测试覆盖最近三次趋势、冷启动、单档调重、未完成语义、伤病过滤和恢复冲突。Repository 仅负责读取快照、调用候选生成器、校验、持久化和刷新状态。

替代方案是把规则继续写进 `FitnessRepository`。拒绝该方案，因为当前 repository 已承担大量协调职责，规则难以做确定性 JVM 验证。

### 4. 本地规则是最终裁决者

联网 AI 和本地候选生成器都输出同一个 `WeeklyPlanCandidate`。本地约束校验器过滤伤病动作、检查场地器械和重量档位、检查连续训练日肌群恢复，并返回结构化失败原因。AI 文本不能绕过校验。

### 5. 规范化输入快照并计算 SHA-256

生成草稿时对档案、目标、日程、场地器械、伤病、例外、偏好和已完成训练摘要做稳定排序，再编码为规范 JSON 并计算 SHA-256。确认草稿前重新计算；不一致时标记 `stale` 并拒绝确认。

### 6. 整周确认使用单个 SQLite transaction

确认时在一个事务中写入该周全部 `planned_workout` 和 `planned_exercise`，随后将草稿标为 confirmed 并推进周期周次。任何写入失败都回滚。

### 7. 六个 stacked PR 按依赖顺序交付

1. 规格与决策文档。
2. SQLite、实体、Store、迁移与备份。
3. 纯规则引擎。
4. Repository 滚动工作流。
5. 配置与草稿 UI。
6. 训练反馈闭环与整体验证。

每个 PR 以之前 PR 分支为 base；合并时按顺序进入 `main`。

## Risks / Trade-offs

- [自由文本伤病映射误判] → 默认保守排除，展示原因，恢复需二次确认，伤病文本变化使例外失效。
- [重量档位默认值与真实器械不符] → 默认目录可编辑，并按场地分别保存。
- [stacked PR 合并顺序错误] → PR 描述明确依赖和 base，后续 PR 不复制前序提交。
- [草稿快照误判过期] → 规范排序、显式字段、确定性哈希和单元测试。
- [数据库迁移或备份丢数据] → 9→10 设备迁移测试、版本 4 导入测试和版本 5 全量 round-trip。
- [规则与 AI 输出冲突] → 规则校验结果优先；冲突时拒绝草稿并展示结构化原因。
- [首轮改动范围过大] → 按六个 PR 拆分，UI 最后接入，前序 PR 只暴露经过测试的窄接口。

## Migration Plan

1. PR 2 将数据库升级到 10，仅新增表/列并验证 9→10。
2. 旧用户首次进入时尚无活动周期；现有正式计划和训练历史保持不变。
3. 用户创建首个周期后才写入新表。
4. 备份导入先校验版本，再在单事务中恢复；版本 4 使用新字段默认值。
5. 若应用版本回滚，旧 APK 不读取新表，但原有表仍可工作；新数据应通过版本 5 备份保留。

## Open Questions

无产品级未决问题。近期历史的内部权重、默认器械档位目录和伤病映射词典作为实现细节，在对应 PR 中通过显式常量与测试固定。
