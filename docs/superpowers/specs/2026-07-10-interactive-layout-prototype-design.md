# i fitness 单一主任务交互原型设计

## 1. 决策摘要

本次先制作一个浏览器内可交互的移动端设计原型，不直接修改 Android/Compose 工程。视觉基准采用三张候选图中的第一张“单一今日任务”方向，并保留现有 App 的橙色主操作、米白页面、深色训练 Hero、绿色进度语义和本地动作 GIF。

原型用于验证三件事：

1. 五项一级导航是否比当前七项导航更清楚。
2. 首页只展示一个主任务后，开始或继续训练是否足够直接。
3. 训练记录闭环能否在一个沉浸页面内完成，避免关键操作被 GIF 和全局导航挤出首屏。

## 2. 目标与非目标

### 目标

- 将底部导航固定为：首页、计划、训练、饮食、我的。
- 首页根据状态只展示一个主任务：开始训练、继续训练或查看完成结果。
- 打通“首页开始训练 → 记录本组 → 休息 → 下一组/动作 → 结束训练 → 首页进度更新”的可交互闭环。
- 让计划、饮食、个人资料和动作库具备足以验证布局的真实交互。
- 在 390×844 的移动视口内完成主要操作，无横向滚动，主要按钮始终容易触达。
- 使用浏览器本地状态保存交互结果，刷新后仍能看到训练与饮食变化。

### 非目标

- 不在这一阶段修改 Kotlin、Compose、SQLite 或 Android 导航代码。
- 不接入 Figma、云服务、账号体系或社交能力。
- 不发送真实 AI 请求，不录入或保存 API 密钥。
- 不重做 1324 个动作的完整生产级动作库，只提供搜索、筛选和详情的代表性交互。
- 不把原型当成正式 Android 验收结果；Android 迁移后仍需 Gradle、APK 和模拟器验证。

## 3. 产品结构

### 一级导航

| 入口 | 职责 | 主要动作 |
|---|---|---|
| 首页 | 今日任务、本周进度、快捷入口 | 开始/继续训练 |
| 计划 | 周历、本周安排、计划详情 | 查看或编辑计划 |
| 训练 | 今日训练准备 | 开始训练 |
| 饮食 | 今日营养摘要和饮食记录 | 添加一餐 |
| 我的 | 档案摘要、偏好、场地、智能设置、备份 | 进入二级设置 |

“动作库”从一级导航下沉为首页快捷入口、训练准备页入口和计划编辑页入口。“智能”不再作为一级页面：计划生成放在计划页，动作替换放在训练页，饮食估算放在饮食页，接口配置放在“我的 > 智能设置”。

### 二级页面与浮层

- 动作库：搜索、肌群筛选、动作详情。
- 计划详情：使用全屏二级页面查看动作、调整日期与训练目标。
- 添加饮食底部弹层：拍照估算入口、手动记录入口。
- 编辑档案、场地和智能设置：分别使用全屏二级页面，顶部提供明确返回入口。
- 训练总结：使用全屏结果页完成确认并返回首页。

## 4. 页面设计

### 首页

首页默认假设用户已完成首次配置，不再显示“首次配置”卡片。页面顺序固定为：

1. 顶部问候与当前场地。
2. 唯一深色训练 Hero。
3. 本周进度与周视图。
4. 记饮食、动作库两个轻量快捷入口。
5. 五项底部导航。

Hero 主标题使用计划名“胸部力量 A”，动作“史密斯机卧推”作为缩略图与副信息。空闲状态显示“开始训练”；存在未结束训练时显示“继续训练”；训练完成后显示完成摘要和下一次训练日期。首页不会同时出现开始、继续和首次配置三个主按钮。

### 计划

页面先展示周视图和本周安排，再展示低频的月计划生成功能。日期卡明确区分训练日与休息日，不使用“空”作为状态文案。点击计划进入详情；“新计划”打开轻量编辑浮层；月计划生成只展示本地草稿结果，不调用真实 AI。

### 训练准备与训练中

从首页或训练 Tab 进入训练准备页，展示计划名、预计时长、动作列表和开始按钮。点击开始后进入沉浸式训练页：

- 隐藏五项底部导航。
- 顶部显示返回/暂停、计划名、已用时间和组进度。
- 动作切换放在紧凑横向列表。
- GIF 使用紧凑 4:3 区域，可点击展开，不占满首屏。
- 重量、次数、体感和“完成本组”在首屏可见。
- “完成本组”固定在底部拇指区。
- 完成一组后进入休息状态，显示倒计时、跳过休息和下一组信息。
- 完成当前动作目标组数后自动切换到下一个动作。
- 用户可主动结束训练；未完成时要求二次确认。

训练结束后展示总结页：完成组数、总容量、训练时长和体感分布。确认总结后回到首页并更新本周完成次数。

### 饮食

饮食页首先展示今日热量、蛋白质、碳水和脂肪摘要，然后展示按时间排序的饮食记录。页面只有一个“添加一餐”主入口；点击后从底部弹层选择“拍照估算”或“手动记录”。拍照估算只演示选择图片、生成草稿和确认入库状态，不发送图片。手动记录保存后立即更新今日摘要与时间线。

### 我的

首屏展示只读档案摘要，而不是完整编辑表单。下方使用设置列表：训练偏好、场地与器械、智能设置、数据备份、关于。点击“编辑资料”再进入表单。智能设置只演示统一的连接状态，不出现“已连接”和“密钥未保存”并存的冲突。

### 动作库

动作库采用搜索框、肌群筛选和一个分组列表。整行可点击，不重复放置“查看”按钮。动作详情展示 GIF、目标肌群、器械、动作提示与“用于本次训练”。原型只加载少量代表动作，但保持与实际 1324 动作数据的字段一致。

## 5. 视觉与交互规范

- 移动画布：390×844；桌面浏览器只提供中性的居中预览环境，不绘制厚重手机外壳。
- 页面背景：暖米白；正文使用深墨色；次要文字采用满足普通文本对比度的深灰。
- 橙色只用于主操作；绿色只用于完成、进度和已选状态；蓝色不再承担主要状态。
- 圆角三档：大模块 24px、普通容器 16px、按钮和输入框 14–16px。
- 正文 14–16px；页面标题 28–32px；不在同一屏大量使用超黑字重。
- 所有交互目标至少 48×48px。
- 普通列表优先使用分组、留白和分割线，不把每一行都做成独立卡片。
- 页面切换使用 180–260ms 淡入/位移；支持 `prefers-reduced-motion`。
- 图标使用 Material Symbols Rounded；若字体资源加载失败，保留可读文字标签，不能以 CSS 图形代替。动作图使用项目现有 GIF/缩略图，不使用 CSS 画图或占位符伪造资产。

## 6. 原型技术设计

### 文件与复用策略

在现有 `.scratch/fitness-app-mvp/健身App-主流移动端交互重设计.html` 的基础上创建独立新原型，避免破坏历史方案。目标文件：

```text
.scratch/fitness-app-mvp/健身App-单一主任务交互原型.html
```

原型继续使用单文件 HTML + CSS + 原生 JavaScript，不引入构建工具。这样可以直接通过本地 HTTP 服务预览，也便于与现有设计稿对照。

### 状态模型

```text
route: home | plan | training-prep | training-active | food | profile | library | detail
workoutStatus: idle | active | resting | completed
currentExerciseIndex: number
completedSets: number
targetSets: number
weightKg: number
reps: number
feeling: 轻松 | 合适 | 吃力
restRemainingSeconds: number
weeklyCompleted: number
foodEntries: list
profile: object
```

状态写入浏览器 `localStorage`，键名固定为 `ifitness-layout-prototype-v1`。原型提供“重置演示数据”入口，方便重复测试不同状态。路由使用 URL hash，使刷新后能回到当前一级页面；训练中的临时倒计时根据时间戳恢复，不依赖持续运行的定时器。

### 事件流

```text
首页开始训练
  → 训练准备
  → 开始
  → 训练中
  → 调整重量/次数/体感
  → 完成本组
  → 休息倒计时
  → 下一组或下一动作
  → 结束训练
  → 训练总结
  → 首页进度更新
```

计划、饮食和个人资料的保存只修改原型本地状态，并通过页面内反馈或轻量 Toast 告知结果。所有 AI 相关动作返回预设草稿，明确标记“演示结果”。

## 7. 错误、空状态与恢复

- 动作图片加载失败：保留动作名和训练控制，显示“动图暂不可用”，不阻塞记录。
- 本周无计划：首页主按钮改为“创建本周计划”，跳转计划页。
- 训练中离开：保留进行中状态，首页 Hero 改为“继续训练”。
- 手动饮食输入不完整：在字段附近显示错误，不清空已输入内容。
- localStorage 不可用：退化为当前会话内存状态，并提示刷新后不会保留。
- 重置演示数据：需要二次确认，避免误清空测试状态。

## 8. 验收标准

### 核心交互

- 五个底部入口均可点击，选中态与页面内容一致。
- 首页始终只有一个训练主按钮。
- 开始训练后底部导航消失，“完成本组”在 390×844 首屏可见。
- 重量、次数和体感均可操作，完成组数能正确递增。
- 休息倒计时可跳过，并能进入下一组或下一动作。
- 结束训练后首页本周完成次数和训练状态更新。
- 饮食记录保存后今日摘要更新。
- 动作库支持搜索、筛选和查看详情。
- 刷新页面后训练、饮食和当前一级路由能够恢复。

### 视觉与可用性

- 390×844、430px 宽和桌面预览均无横向滚动。
- 底栏固定为五项且文字不截断。
- 所有主要按钮和自定义 Chip 的触控区域不小于 48px。
- 次要正文对比度不低于 4.5:1。
- 页面不会出现卡片嵌套、重复主按钮或无功能的装饰性顶部图标。
- 开启减少动态效果后，不依赖动画才能理解状态变化。

### 验证方式

- 使用本地 HTTP 服务打开原型。
- 在 390×844 和 430px 宽视口逐页截图。
- 用浏览器自动化完成训练闭环、饮食保存、导航切换和刷新恢复。
- 检查控制台无错误、无未处理 Promise、无资源 404。

## 9. Android 迁移边界

原型通过后再把信息架构与交互迁移到 Compose。迁移时保留当前本地 SQLite、Repository、GIF、计划和记录逻辑，只调整导航、页面组合与显示状态。Android 阶段必须重新执行单元测试、APK 构建、模拟器安装、真实交互截图和必要的数据库状态核对；浏览器原型通过不等于 Android 实现完成。
